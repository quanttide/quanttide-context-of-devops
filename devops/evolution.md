# 演化路径与边界

> 来源：工作日志提炼，跨越 06-19 至 07-04 的持续迭代

## 一、意外发现：DevOps 作为自动化核心底座

DevOps 云在体系中的位置从一开始没有预料到——它成了最基础的地基式领域。原因有二：

1. **容易构建**：规则明确，边界清晰，天然适合自动化
2. **天然分割上下文**：把一个大型软件的版本管理切成小问题，为其他领域留下可操作的上下文粒度

这一发现改变了架构认知：**实际运行中的关系与设计时的预想完全不同。做出来是什么样，就是什么样——工程实践优先于架构预设。**

## 二、Skill 组合与自动化发布流

从原子能力到组合工作流的演进路径：

```
单个 Skill（如 cos blueprint）
    → Skill 组合
    → 完整自动化发布流
    → 内置到系统成为默认能力
```

这使得量潮云开始具备通用性——它能干很多杂事，成为一个日常工作的默认入口。所有应用层理论上都能做所有事，只是擅长的方向不同：

| 入口 | 定位 |
|------|------|
| 量潮管理后台 | 擅长内部流程封装（规则引擎） |
| 量潮云 | 平衡通用，覆盖杂项工作 |
| 量潮数据 | 专注数据业务 |

## 三、应用/包的单仓模式与契约固化

在 monorepo 实践中提炼出两种内置模式：

- **application（应用模式）**：完整的可部署应用
- **package（库模式）**：可复用的库

这些模式趋向于写入章程作为默认规范，并在代码库中内置实现。不必追求 1:1 复刻，将常用逻辑固定下来反而让系统更可靠。

## 四、积木式子系统组合：云原生 + SaaS/PaaS + AI 原生

一种全新的云厂商形态正在显现：

- **硬工程层**：提供基础原子能力
- **积木式组合层**：子系统级别的排列组合，可以生成远超规划的用途
- **AI 原生入口**：所有应用层既是通用平台，又有不同的调参倾向

这是**云原生基础设施 + SaaS/PaaS 分层 + AI 原生交互**的三位一体，是真正面向 AI 时代的云架构。

## 五、持续暴露的新边界

近期日志中暴露的几个需要关注的边界问题：

1. **AI 生成 Changelog 的不稳定性**：大模型引入后可能重复标题或生成错误标题，需要在提示词和程序层面双重排查
2. **自举实验**：用新版本发布旧版本、让流水线自我接管，是验证平台成熟度的关键实验
3. **事实源扩展**：除了 git，配置文件、特殊文件（如 changelog）也应纳入事实源体系

## 六、跨仓库协作模式

DevOps 云和资产云的配合方式：

```
资产云（归档治理成果）
    → DevOps 云（操作 Git 仓库）
    → 同步到各端
```

- 资产云负责归档，DevOps 云负责操作
- 资产云可以直接导入 DevOps 库操作 Git 仓库，简化逻辑
- 复杂的组合步骤放在量潮云编排层处理

## 七、后续建议

2. **固化 review.md → blueprint.md 循环**：把"强制注入上下文"机制做成标准化 Skill 或命令
3. **启动测试环节**：发布环节已经成熟，下一步把测试纳入体系（AI 根据 Minor 提交自动生成测试用例并执行）
4. **自举链路**：用 DevOps 工具发布 DevOps 工具本身，完成从"造工具"到"工具造工具"的质变
