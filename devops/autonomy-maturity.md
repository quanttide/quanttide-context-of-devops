# DevOps 工具自治与版本成熟度

> 来源：工作日志提炼，跨越 06-19 至 07-04 的持续迭代

## 一、版本管理的智能决策

自动发版的权限设计，遵循"安全第一，逐步自治"的原则：

| 版本类型 | 权限 | 理由 |
|----------|------|------|
| **Major（大版本）** | 必须人类外部传入，AI 绝对禁止越权 | 防止体系性重构意外破坏生产 |
| **Minor/Patch（小版本）** | AI 自主判断 | 0.x 阶段快速迭代，1.x 阶段严格保护 |

**工作流闭环：**

```
status（扫描现状）
    → doctor（智能体检）
    → AI 决策版本号
    → 生成 changelog
    → 发布
```

这个链路打通后，用 Skill 封装成一条上层命令，DevOps 的基础自治就成型了。

**版本决策逻辑：**
- 如果给定版本号 → 直接使用
- 如果未给定版本号 → AI 自主判断（暂时限制不能发 major）
- 0.x 阶段：minor 是大的版本，patch 是小的版本，主要看是否有 break change
- 1.x 阶段：需要更严格措施，不能破坏现有生产逻辑

## 二、契约、Scope 与积木块化

**Scope 的显现：** DevOps 契约的出现（如 Scope 定义仓库划分），意味着工具正在从"执行脚本"向"规范引擎"转变。

**文档即代码的两级约束：**

- **章程**：已确定的标准，作为系统的最高约束
- **手册**：工具使用方式和已知问题，作为 AI Skill 的底稿

这两者形成飞轮：手册越清晰，AI 执行越准确；AI 执行中发现的新边界，又反过来更新手册。

## 三、版本成熟度模型

从迭代中提炼出的版本阶段定义：

| 版本 | 定义 | 标志 |
|------|------|------|
| 0.1 | 最小可用 | 上线第一个功能 |
| 0.5 | 初步生产可用 | 大框架成型，结构稳定 |
| 0.7 | 细节收敛中 | 大框架清晰，但内部细节仍混乱不透明 |
| 1.0 | 内部收敛完成 | API 稳定，内部场景已基本穷尽，再迭代只能向外 |

**关键认知：**
- 1.0 之前都属于内源阶段
- 1.0 不一定代表对外稳定版本，可以定义为"只在自己生态内使用"的公开测试
- **内部收敛完成是发 1.0 的充分条件**

**关于版本节奏：** 0.5 版本差不多开始初步生产可用，0.1 只是有一两个功能。一个平台要搭成完整框架，至少得 0.5。0.10 左右大框架才算比较稳定。

## 四、status / doctor 命令体系

`status` 命令用于扫描现状。在此基础上可以扩展 `doctor` 命令进行智能检查——不仅能看状态，还能给出"诊断建议"，这会让工具具备"顾问"属性。

## 五、发布能力是核心基础设施

发布这个环节已经成熟，接下来就可以进入下一个环节——测试。每个环节都可以类似地构建，一旦成型，就拥有了规范化和自动化代码的工具，整个系统的有效性会极大提高。

**自传播触发点：** 当命令行简化到手册几行能写完，Changelog 能写清楚时，就开始具备自传播性。这意味着工具从 Alpha 阶段进入了 Beta 阶段。
