# 架构设计与工程哲学

> 来源：工作日志提炼，跨越 06-19 至 07-04 的持续迭代

## 一、治理驱动重构

"所有云的思路都清晰了，都是治理工具。"

无论是产品云、资产云还是 DevOps 云，本质都是对散乱信息、过程和资产的**治理**。通过各个领域的 Profile 记录治理成果，系统就能沉淀知识并不断升级。

## 二、跨中心部署与合规性

**架构决策：** DevOps 核心放在国外（直接调用 GitHub API），国内只做信息同步。这既利用了国外生态的完整性，又规避了国内直接调用可能面临的网络和合规风险。

**数据流转模型：**
```
资产云（归档）
    → DevOps 云（操作）
    → Git 仓库
```

资产云可以直接导入 DevOps 库操作 Git 仓库，简化逻辑。

**合规策略：** 目前体量小，先用内部接口保证可用；未来用法务档案管理跨境数据传输的合规问题。先跑通业务，后补齐法务。

## 三、分层架构：编排层设计

将底层基础工具（严格规范下执行、保障安全）与上层组合操作分离：

| 层级 | 角色 | 特征 |
|------|------|------|
| 底层工具 | 原子化操作 | 严格规范、安全、为量潮体系客户提供便捷 |
| 量潮云（编排层） | 排列组合原子工具 | 处理多步骤复杂任务，保证鲁棒性 |

## 四、从单体工具到操作系统内核的蜕变

DevOps 工具的演进轨迹，完美复刻了操作系统的诞生过程：

| OS 类比 | DevOps 对应 | 说明 |
|---------|-------------|------|
| 系统调用层 | 原子操作（如资产云接口） | 直接操作 Git 仓库 |
| 标准库层 | status、doctor、发版判断 | 严格规范，保证安全 |
| Shell/编排层 | 量潮云 | 排列组合底层工具 |
| 用户程序层 | Skill 封装的上层命令 | 对外交互接口 |

Scope 的出现，就相当于这个内核里开始有了"进程边界"和"命名空间"的概念。

## 五、认知摩擦的消除

**演化路径：** 执行摩擦（环境配不通、代码跑不起来）→ 认知摩擦（AI 是否理解了人类的意图）。

通过 GitHub Issue 传递机制，AI 能够自主定位 Bug 并修复，人类只需引导长远设计。这说明 AI 的认知正在向人类收敛。

**标志性感受：** "感觉到解决问题开始变得顺滑"、"脱离了老是在当前问题打转的状态"——这正是工程体系从"作坊式"走向"平台化"的典型特征。

## 六、边界是试出来的，不是想出来的

AI 原生工具的边界与传统软件完全不同。AI 有时聪明得惊人，有时又笨得离谱。扬长避短的唯一方法就是在实践中碰撞出这些边界，然后写进手册固定下来。

> 实际总结能直接看到边界感，这是设计不出来的。

## 七、动作优先于讨论

**决策：** 先纳入 PR（确定的动作），暂缓 Issue（讨论）。

**深层逻辑：** 在构建自动化体系时，坚决不让"噪音"过早进入系统。很多团队做 DevOps 自动化失败，就是一头扎进 Issue 的 NLP 分类泥潭里。现阶段流程的"形式化"比过程的"自动化"更重要。先跑通确定的链路，再逐步消化不确定的输入。

## 八、自传播性

当一个命令行工具的 Manual 可以用几行写完，Changelog 清晰明了时，它就从 Alpha 阶段进入了 Beta 阶段。

**PMF 核心：** 接入摩擦趋近于零。开发者是最没有耐心的群体。把"几行手册"作为内测的门槛，意味着所有复杂性都吞进了系统内部（由 AI 和底层工具消化），留给用户的只有极简的交互。

这是工具能够从"自用"走向"开源/商业化"的唯一路径。
