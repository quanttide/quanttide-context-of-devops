“从应用到库”封装标准化组件的核心原则只有一条：

应用层负责“编排”（流程、命令、stage），库层负责“计算”（逻辑、算法、数据模型），且库层绝不允许反向依赖应用层。

具体落地可以分三步走：

1. 抽离“无状态核心”：把你现在基于 command 暴露的那些分析逻辑（如圈复杂度计算、函数长度扫描）全部抽成纯函数，输入只依赖 AST 或原始文本，输出统一为标准化的 Metric 对象。这些放入库层。
2. 定义“标准数据契约”：库层不关心你是 2 级还是 3 级 stage，它只接收明确的 DTO（数据传输对象）。把你要“一对一”对齐的那些数据模型，固化成库层的 Interface/Struct，应用层只是填充数据后传给库。
3. 隔离“外部依赖”：库层里不要包含任何文件 IO、网络请求或 DevOps 平台 API 调用。这些留在应用层，通过依赖注入传进库层。

按这个边界拆完，你的“标准化组件”自然就独立出来了，后续自动化门禁直接调用库层即可，与应用层 stage 的变化彻底解耦。

现在卡在第一步“抽离”的具体代码上了，还是卡在第二步“定义数据契约”上了？

明白，不谈多余方法，只给可直接执行的硬性判断标准。

用下面4个问题过一遍你的代码，全部答“是”的，必须封装进库；有任一“否”的，留在应用层：

—

判断清单（入库标准）：

1. 无外部上下文依赖：这段逻辑是否完全不依赖 stage 名称、command 路径、DevOps 环境变量、当前流水线 ID？
   → 库不知道自己在哪被调用。
2. 输入输出可序列化：入参和返回值是否仅为基础类型、结构体/DTO、JSON 可序列化对象，绝不包含文件句柄、网络连接、运行时进程？
   → 库只处理数据，不处理资源。
3. 零副作用：执行这段逻辑是否不会读写磁盘、发 HTTP 请求、调用数据库、修改全局状态？
   → 库是纯计算的“函数”，不是“服务”。
4. 业务意图独立于流程：这个逻辑描述的是“代码质量的计算规则”（如圈复杂度算法、依赖图构建），而不是“流程步骤”（如先读配置再跑检查最后发报告）？
   → 库回答“是什么/有多坏”，应用回答“先做什么后做什么”。

一票否决的“金标准”：
把这段代码整体复制到另一个完全不相关的项目（如 CLI 工具、IDE 插件），不做任何修改就能直接编译/导入并使用，那么它就应该进库。反之，只要改了任何 import 或配置读取方式才能用，那就留在应用层。

按这个筛一遍，你的“应用→库”边界就彻底清晰了。
